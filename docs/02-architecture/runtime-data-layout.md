# Runtime data layout

lipost keeps the repo read-only — **all runtime state lives outside it**, under your home directory. The
repo ships only the code (`bin/lipost`), the baked style/prompt examples (`share/`), and the installer.

Two roots, following an XDG-ish convention on macOS:

- `~/.config/lipost/` — **configuration and history** (credentials, tokens, bot settings, post log).
- `~/.local/share/lipost/` — **working data** (the draft queue, staged images).

Logs go to `~/Library/Logs/lipost.log`.

---

## The file map

| Path | Purpose | Notes |
| --- | --- | --- |
| `~/.config/lipost/auth.json` | LinkedIn `client_id` / `client_secret` | `chmod 600` |
| `~/.config/lipost/token.json` | OAuth access token + member URN | `chmod 600` |
| `~/.config/lipost/bot.json` | Bot settings | `chmod 600`; see [Bot config reference](../03-cli-reference/bot-config-reference.md) |
| `~/.config/lipost/state.json` | Bot state — `last_run_at`, `paused` | |
| `~/.config/lipost/posts.jsonl` | Append-only post history (URN + image) | one JSON object per line |
| `~/.config/lipost/prompt.md` | Live persona prompt (article + image generation) | seeded from `share/prompt.example.md` |
| `~/.config/lipost/user_style.md` | Optional user additions to the style layer | appended after `share/style.md` |
| `~/.config/lipost/launchd/` | Generated plists | symlinked into `~/Library/LaunchAgents/` |
| `~/.local/share/lipost/drafts/` | Draft queue — one directory per draft | each holds `meta.json` (+ image) |
| `~/.local/share/lipost/images/pending/` | Image staging — awaiting generation | drained by the generate phase |
| `~/.local/share/lipost/images/skipped/` | Images Claude declined | |
| `~/.local/share/lipost/images/generated/` | Article images generated by OpenAI | |
| `~/Library/Logs/lipost.log` | Bot logs + lipost stdout/stderr | tail with `lipost bot logs -f` |

`REPO_DIR` (and thus `share/`) is resolved from the script's own path, so the baked `style.md` /
`prompt.example.md` are found regardless of where the `bin/lipost` symlink lives.

---

## Permissions

The three files that hold secrets — `auth.json`, `token.json`, `bot.json` — are written `chmod 600`
(owner-only). Don't relax this; `auth.json` holds your LinkedIn app secret and `token.json` holds a live
member token.

---

## The draft directory

Each draft is a directory under `drafts/` named with a timestamped slug (e.g.
`20260430T093412_a1b2c3/`) containing a `meta.json` and, for image drafts, the image file. The `meta.json`
schema is the queue's contract — the cron and the review TUI both read and write it, and you can hand-edit it
to bulk-approve, fix typos, or change status:

```json
{
  "slug": "20260430T093412_a1b2c3",
  "status": "pending_approval",
  "caption": "...",
  "alt_text": "...",
  "image_filename": "image.png",
  "source_filename": "diagram.png",
  "source_url": null,
  "created_at": "...",
  "approved_at": null,
  "rejected_at": null,
  "posted_at": null,
  "urn": null,
  "model": "claude-opus-4-7"
}
```

The cron **trusts what's on disk** — there is no separate index. Status moves
`pending_approval → approved → posted` (or `→ rejected`).

## Legacy-path migration

`lipost init` and `lipost bot init` detect and migrate older installs, interactively and idempotently:

- `~/.config/linkedin-cli/` → `~/.config/lipost/`
- `~/.config/lipost-bot/` → `~/.config/lipost/`
- `~/code/lipost-bot/{drafts,images,prompt.md,user_style.md}` → `~/.local/share/lipost/` + `~/.config/lipost/`

Legacy paths are also read as read-only fallbacks before re-init, so existing installs keep working until you
migrate.
